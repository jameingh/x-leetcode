# 第二阶段：层级结构的拆解艺术 —— 二叉树与递归 (Binary Tree & Recursion)

如果说数组和链表是“一条线”（线性结构），二叉树就是“一张网”或“一个家族谱”（层级结构）。

## 1. 核心思维：什么是递归？ —— “分身术”模型

很多人觉得递归难，是因为总是试图在脑子里追踪每一层调用的细节。这会让你大脑宕机。

**思维模型**：
想象你是一个部门主管，要统计全公司的员工人数。
- **递归写法**：你不需要自己去数。你只需要问你的两个下属（左经理和右经理）：“你们各自带了多少人？”
- **汇总逻辑**：全公司总人数 = 左经理的人数 + 右经理的人数 + 你自己(1)。
- **边界条件**：如果某个经理手下没人了（空节点），他会告诉你结论是 0。

**递归三要素（必背）**：
1.  **确定返回值和参数**：你需要下属给你汇报什么结果？
2.  **确定终止条件**：到哪一层就不用往下问了？（比如遇到空节点）
3.  **确定单层逻辑**：每个人拿到下属的汇报后，该怎么加工并传给上司？

### 💡 思维对齐：设计层 vs 执行层

当你坐在电脑前时，通过下面的对照表，你可以瞬间将“设计方案”转化为“稳健代码”：

| 递归三要素 (设计层) | 你的代码注释 (执行层) | 这一步的物理意义 |
| :--- | :--- | :--- |
| **1. 确定返回值与参数** | **`def func(root, target):`** | **定契约**：主管与下属约定好汇报内容与所需资料。 |
| **2. 确定终止条件** | **`# 终止条件：空节点`** | **设底线**：规定哪一层不准再问下去，防止大脑宕机。 |
| **3. 确定单层逻辑** | **`# 递归调用`** | **分任务**：主管处理一小部分，剩下的丢给分身（递归）。 |
| (业务细节) | **`# 叶子节点逻辑`** | **看收官**：在搜索类题目中，真正的结算点往往在此。 |

## 3. 标准解题模板

处理二叉树问题，通常有两类策略：

### A. 深度优先搜索 (DFS / 递归)
利用递归，深入到底再回溯。
- **前序遍历**：根 -> 左 -> 右（先处理自己，再处理孩子）
- **中序遍历**：左 -> 根 -> 右（常用于二叉搜索树）
- **后序遍历**：左 -> 右 -> 根（常用于需要孩子结果来决定自己的场景）

### B. 广度优先搜索 (BFS / 层序遍历)
像手电筒横扫一样，一层一层处理。通常使用 **队列 (Queue)** 辅助。

## 6. 递归设计的进阶思考：为什么是“余额扣减”？

在设计递归（尤其是像 **112. 路径总和** 这样的搜索题）时，有两种思维模型：

### A. 累加模型 (Bottom-Up 心态)
- **想法**：“我要带着目前的总和，每走一步加一下，最后看等不等于目标。”
- **代价**：通常需要引入额外的变量或参数来维护这个“当前状态”。

### B. 余额扣减模型 (Top-Down 心态) —— **推荐！**
- **核心逻辑**：**问题规模缩减 (Problem Reduction)**。
- **推演**：如果要找路径和为 `22` 的路径，且当前节点值是 `5`。那么剩下的任务就是：**在子树中找一条和为 `17` ($22-5$) 的路径**。
- **优势**：
    1. **参数复用**：直接利用题目给的 `targetSum` 变量记录“剩余任务量”，代码极简。
    2. **专注局部**：每一层主管只关心“我这一层消耗了多少”，剩下的丢给下属，逻辑上更符合分治的思想。

**结论**：当你面对一个“全局目标”并需要向下分发任务时，优先思考“我这一层能消耗掉多少目标”，而不是“我从上面累积了多少”。

## 7. 学术语对照：大纲里的关键词都在哪？

你在 **`syllabus.md`** 大纲中看到的那些“高大上”的词汇，其实早已悄悄融入了你的代码中。看下面的映射表，你就能瞬间秒懂：

| 大纲关键词 | 我们的实战模型 | 对应你代码中的动作 |
| :--- | :--- | :--- |
| **状态传递** | **携带余额 / 向上汇报** | **参数**（下放状态）或 **返回值**（带回结果）。 |
| **归约 (Reduction)** | **汇总逻辑** | 拿到左右子树结果后的 **加工步骤**（如：`1 + max(L, R)`）。 |
| **分治思维 (Divide & Conquer)** | **分身术 (部门拆解)** | **核心逻辑**：1、分解 (Divide)：对应代码里的 root.left 和 root.right。你把一棵大树，“分解”成了两个更小的部门。2、解决 (Conquer)：对应代码里的递归调用。你把小部门丢给分身去解决。3、合并 (Combine)：对应代码里的汇总逻辑。不管是 1 + max(L, R) 还是 L + R + 1，你都是在把分身的答案“合并”成最终结果。 |

### 💡 深度理解“归约”
“归约”这个词听起来很玄，但你只要记住：**当你把“整棵树的大问题”变成“两个子树的小问题”并最终汇总成一个数时，你就在进行归约。** 这就是你写 `left + right + 1` 的本质。
